## Commenting Standards

### Purpose of Comments
- Explain **WHY**, not just **WHAT**
- Document complex logic and algorithms
- Clarify non-obvious decisions
- Warn about gotchas or edge cases
- Link to relevant documentation or tickets

### Comment Quality
```python
# Bad: Increment counter
counter += 1

# Good: Track number of failed retries for exponential backoff
retry_count += 1

# Bad: Check if x is 5
if x == 5:

# Good: API returns 5 for "pending" status, needs special handling
if status_code == 5:
```

### TODO Comments
Include date and initials for future work:
```python
# TODO (2024-02-21, ccharland): Refactor to use async/await for better performance
# FIXME (2024-02-21, ccharland): Handle edge case when list is empty
# HACK (2024-02-21, ccharland): Temporary workaround until API v2 is available
```

### Avoid Over-Commenting
```python
# Bad: Too obvious
x = 5  # Set x to 5
name = "John"  # Set name to John

# Good: Self-documenting code
max_retry_attempts = 5
user_display_name = "John"
```

## File Organization Standards

### Naming Conventions
- **Files**: kebab-case (`user-management.py`, `deploy-stack.sh`)
- **Classes**: PascalCase (`UserManager`, `CloudWatchAlarm`)
- **Functions/Variables**: snake_case (`get_user_data`, `retry_count`)
- **Constants**: UPPER_SNAKE_CASE (`MAX_RETRIES`, `API_ENDPOINT`)
- **PowerShell exception**: `.ps1` and `.psm1` files follow `Verb-Noun` PascalCase convention per PowerShell standards (`Get-Something.ps1`, `Lambda-Customizations.psm1`)
Remind user to use a singular noun.

### Directory Structure
- Organize by function/feature, not by file type
- Keep related files together
- Use descriptive directory names
- Include README.md in every directory

```
# Good: Organized by feature
project/
├── user-management/
│   ├── __init__.py
│   ├── user.py
│   ├── permissions.py
│   └── tests/
├── billing/
│   ├── __init__.py
│   ├── invoice.py
│   └── tests/

# Bad: Organized by file type
project/
├── models/
│   ├── user.py
│   ├── invoice.py
├── controllers/
├── tests/
```

### File Headers/comment based help required

Every script/template must include:
- Purpose/description
- Author and creation date
- Kiro attribution
- Usage instructions
- Prerequisites
- Example usage

See `file-headers.md` for detailed templates.

## Version Control Standards

### Commit Messages
Follow conventional commits format:
```
<type>: <description>

[optional body]
[optional footer]
```

Types: `feat`, `fix`, `docs`, `refactor`, `test`, `chore`

See `git-workflow.md` for detailed guidelines.

### Commit Frequency
- Keep commits atomic - one logical change per commit
- Commit after completing each unit of work
- Don't wait for perfection
- Each commit should leave code in working state

### Branch Strategy
- Use descriptive branch names: `feature/alarm-management`
- Keep branches short-lived (days, not weeks)
- Merge frequently to avoid conflicts
- Delete branches after merging
- Encouage user to avoid scopoe creep

## Documentation Standards

- Document all public APIs
- Include parameter types and return values
- Provide usage examples
- Document exceptions/errors

### Documentation Maintenance

## Security Standards

### Never Commit Secrets
- No API keys, passwords, or credentials in code
- Use environment variables or parameter stores
- Add sensitive files to .gitignore
- Use AWS Secrets Manager or Systems Manager Parameter Store

### Input Validation
```python
# Always validate and sanitize inputs
def process_user_input(user_id: str) -> dict:
    if not user_id or not user_id.isalnum():
        raise ValueError("Invalid user ID format")
    
    # Sanitize before using in queries
    safe_user_id = sanitize_input(user_id)
    return get_user_data(safe_user_id)
```

### Least Privilege
- Grant minimum necessary permissions
- Use specific resource ARNs, not wildcards
- Implement resource-based policies
- Regular audit of permissions

### Encryption
- Enable encryption at rest by default
- Use TLS/HTTPS for data in transit
- Use AWS KMS for key management
- Rotate credentials regularly

### Secure Defaults
```yaml
# Good: Secure by default
Resources:
  MyBucket:
    Type: AWS::S3::Bucket
    Properties:
      PublicAccessBlockConfiguration:
        BlockPublicAcls: true
        BlockPublicPolicy: true
        IgnorePublicAcls: true
        RestrictPublicBuckets: true
      BucketEncryption:
        ServerSideEncryptionConfiguration:
          - ServerSideEncryptionByDefault:
              SSEAlgorithm: AES256
```

## Error Handling Standards

### Always Handle Errors
```python
# Bad: No error handling
def get_user(user_id):
    return dynamodb.get_item(Key={'id': user_id})

# Good: Proper error handling
def get_user(user_id: str) -> dict:
    try:
        response = dynamodb.get_item(Key={'id': user_id})
        return response.get('Item', {})
    except ClientError as e:
        logger.error(f"Failed to get user {user_id}: {e}")
        raise
    except Exception as e:
        logger.error(f"Unexpected error getting user {user_id}: {e}")
        raise
```

### Meaningful Error Messages
```python
# Bad: Generic error
raise Exception("Error occurred")

# Good: Specific, actionable error
raise ValueError(
    f"Invalid alarm ARN format: {alarm_arn}. "
    f"Expected format: arn:aws:cloudwatch:region:account:alarm:name"
)
```

### Logging Errors
- Log errors with sufficient context
- Include relevant identifiers (user_id, request_id)
- Log stack traces for debugging
- Use appropriate log levels

```python
import logging

logger = logging.getLogger(__name__)

try:
    result = process_request(request_id)
except Exception as e:
    logger.error(
        f"Failed to process request {request_id}: {str(e)}",
        exc_info=True,  # Include stack trace
        extra={'request_id': request_id, 'user_id': user_id}
    )
    raise
```

### Exit Codes for Scripts
```bash
#!/bin/bash

# Use meaningful exit codes
SUCCESS=0
ERROR_INVALID_ARGS=1
ERROR_AWS_CLI=2
ERROR_DEPLOYMENT=3

if [ $# -lt 2 ]; then
    echo "Error: Missing required arguments"
    exit $ERROR_INVALID_ARGS
fi

# Exit with appropriate code
exit $SUCCESS
```

## Code Quality Standards

### Follow Language Style Guides
- **Python**: PEP 8
- **PowerShell**: PowerShell Practice and Style Guide
- **JavaScript**: Airbnb Style Guide
- **YAML**: 2-space indentation

### Code Formatting
- Use consistent indentation (4 spaces for Python, 2 for YAML)
- Limit line length to 100 characters
- Use blank lines to separate logical sections
- Align similar code for readability

### Meaningful Names
```python
# Bad: Unclear names
def f(x, y):
    return x + y

# Good: Descriptive names
def calculate_total_cost(base_price: float, tax_rate: float) -> float:
    return base_price * (1 + tax_rate)
```

### Avoid Magic Numbers
```python
# Bad: Magic numbers
if retry_count > 3:
    time.sleep(60)

# Good: Named constants
MAX_RETRIES = 3
RETRY_DELAY_SECONDS = 60

if retry_count > MAX_RETRIES:
    time.sleep(RETRY_DELAY_SECONDS)
```

### Remove Dead Code
- Delete commented-out code (it's in git history)
- Remove unused imports and variables
- Clean up before committing
- Use linters to catch unused code

## Testing Standards

### Test Coverage
- Write tests for all public functions
- Test both success and failure cases
- Include edge cases and boundary conditions
- Aim for 80%+ code coverage

## Performance Standards

### Optimize When Needed
- Profile before optimizing
- Focus on bottlenecks
- Document performance requirements
- Consider trade-offs (readability vs. speed)

### Resource Management
```python
# Use context managers for cleanup
with open('file.txt', 'r') as f:
    data = f.read()

# Close connections properly
try:
    conn = get_connection()
    result = conn.execute(query)
finally:
    conn.close()
```

- Use CloudFormation or Terraform
- Version control all infrastructure
- Test in non-production first
- Document deployment steps

### Deployment Scripts
- Include error handling
- Provide progress indicators
- Log all actions
- Support rollback on failure

- [ ] Tests passing
- [ ] Documentation updated
- [ ] Security review completed
- [ ] Backup/rollback plan ready
- [ ] Monitoring configured
