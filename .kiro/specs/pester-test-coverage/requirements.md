# Requirements Document

## Introduction

This feature adds comprehensive Pester test coverage for all functions in the CharlandCustomizations PowerShell module. The test suite follows the project's existing TEST-PLAN.md guidance, covering public functions (exported via manifest), private helper functions with reusable behavior, and release-support scripts. Tests are organized by functional area, use mocks for external dependencies (AWS, filesystem, git), and target 80%+ code coverage with descriptive test names.

## Glossary

- **Test_Suite**: The complete collection of Pester test files under the `tests/` directory that validate module behavior.
- **Test_Runner**: The Pester framework (v5+) invoked via `Invoke-Pester` to discover and execute test files.
- **Mock**: A Pester `Mock` declaration that replaces a real command with a controlled substitute during test execution.
- **SUT**: System Under Test — the specific function or script being validated by a given test file.
- **Unit_Test**: A Pester test tagged `Unit` that validates a single function in isolation using mocks for external dependencies.
- **Help_Test**: A Pester test tagged `Help` that validates comment-based help discoverability via `Get-Help`.
- **Integration_Test**: A Pester test tagged `Integration` that exercises real external interactions (AWS, git, filesystem) in a sandbox environment.
- **Test_File**: A `.Tests.ps1` file containing one or more `Describe` blocks that validate a specific function or script.
- **Code_Coverage**: The percentage of source lines executed during test runs, measured by Pester's `-CodeCoverage` parameter.
- **AWS_Mock**: A Pester mock for AWS PowerShell cmdlets (e.g., `Get-CFNStack`, `Get-STSCallerIdentity`) that prevents live AWS API calls during unit tests.
- **ShouldProcess_Test**: A test that validates `-WhatIf` and `-Confirm` behavior for state-changing commands.

## Requirements

### Requirement 1: Test File Structure and Organization

**User Story:** As a developer, I want test files organized by functional area mirroring the source structure, so that I can quickly locate and maintain tests for any given function.

#### Acceptance Criteria

1. THE Test_Suite SHALL organize test files into subdirectories matching the source module structure: `tests/Unit/Core/`, `tests/Unit/AWS/`, `tests/Unit/AWS/S3/`, `tests/Unit/AWS/Audit/`, `tests/Unit/CloudFormation/`, `tests/Unit/Git/`, and `tests/Unit/Scripts/`.
2. WHEN a new Test_File is created for a function, THE Test_File SHALL follow the naming convention `<FunctionName>.Tests.ps1`, and WHEN a new Test_File is created for a script, THE Test_File SHALL follow the naming convention `<ScriptBaseName>.Tests.ps1` where ScriptBaseName is the script filename without the `.ps1` extension.
3. THE Test_File SHALL include Kiro attribution as a single-line comment on the first line of the file in the format `# Generated by Kiro, reviewed by <username>`.
4. WHEN a Test_File is created, THE Test_File SHALL dot-source the specific SUT file within a `BeforeAll` block using a path relative to `$PSScriptRoot` (e.g., `. "$PSScriptRoot/../../src/CharlandCustomizations/Private/FunctionName.ps1"`).
5. THE Test_Suite SHALL include a Pester configuration file at `tests/pester.config.ps1` that defines the test run path as `tests/Unit/`, excludes the `Integration` tag from default runs, and specifies code coverage paths targeting all `.ps1` and `.psm1` files under `src/CharlandCustomizations/`.

### Requirement 2: Core Public Function Tests

**User Story:** As a developer, I want unit tests for all root-level public functions, so that I can verify their behavior before release.

#### Acceptance Criteria

1. WHEN `Install-ProfilesFromSource` is invoked with a `-Path` pointing to a directory containing one or more files named after profile types (AllUsersAllHosts, AllUsersCurrentHost, CurrentUserAllHosts, CurrentUserCurrentHost), THE Test_Suite SHALL verify that each matching file is copied to the corresponding `$PROFILE.<ProfileType>` path.
2. WHEN `Install-ProfilesFromSource` is invoked with `-WhatIf`, THE ShouldProcess_Test SHALL verify that no files are written to any `$PROFILE` location.
3. WHEN `Clear-AuthenticodeSignature` is invoked with a file containing a `# SIG # Begin signature block` marker, THE Test_Suite SHALL verify that all content from the signature marker onward is removed and the content preceding the marker remains byte-for-byte unchanged.
4. WHEN `Clear-AuthenticodeSignature` is invoked with a path that does not resolve to an existing file, THE Test_Suite SHALL verify that a non-terminating error is written to the error stream containing the file path.
5. WHEN `Set-FileSignature` is invoked and no valid code-signing certificate is found in the CurrentUser store, THE Test_Suite SHALL verify that a terminating error is thrown with the message "No valid codesign certificate found".
6. WHEN `Set-FileSignature` is invoked with a certificate whose Issuer starts with `CN=Digicert`, THE Test_Suite SHALL verify that the timestamp server is set to `http://timestamp.digicert.com`.
7. WHEN `Set-FileSignature` is invoked with a certificate whose Issuer starts with `CN=Sect`, THE Test_Suite SHALL verify that the timestamp server is set to `http://timestamp.sectigo.com`.
8. WHEN `Invoke-ScriptMultiAccountRegion` is invoked with N profiles and M regions that each authenticate successfully, THE Test_Suite SHALL verify that the ScriptBlock executes exactly N × M times (once per profile-region combination).
9. WHEN `Invoke-ScriptMultiAccountRegion` is invoked with `-IncludeAccountId`, THE Test_Suite SHALL verify that each output object contains an `AccountId` property whose value matches the account ID returned by `Get-STSCallerIdentity` for that profile.
10. WHEN `Update-Powershell7` is invoked on a Windows platform where `Get-Command winget` returns successfully, THE Test_Suite SHALL verify that `winget upgrade Microsoft.PowerShell` is invoked with `--accept-source-agreements`.
11. IF `Set-FileSignature` is invoked with a certificate whose Issuer does not start with `CN=Digicert` or `CN=Sect` and no `-TimeStampServer` is provided, THEN THE Test_Suite SHALL verify that a terminating error is thrown with the message "No Timestamp server could be set, aborting."

### Requirement 3: AWS Function Tests

**User Story:** As a developer, I want unit tests for all AWS-facing functions using mocks, so that I can validate AWS logic without making live API calls.

#### Acceptance Criteria

1. THE Test_Suite SHALL mock all AWS PowerShell cmdlets invoked by the SUT functions (including `Get-CFNStack`, `Get-STSCallerIdentity`, `Get-S3Bucket`, `Get-S3Version`, `Remove-S3Object`, `Get-IAMMFADevice`, `Get-STSSessionToken`, `Use-STSRole`, `Set-AWSCredential`, `Remove-AWSCredentialProfile`, `Start-CFNStackDriftDetection`, `Get-CFNStackDriftDetectionStatus`, `Get-CFNStackResourceSummary`, `Get-CFNStackResourceDrift`, `Get-EC2Region`, `Get-EC2Instance`, `Get-EC2Vpc`, `Get-LMFunctionList`, `Get-S3BucketLocation`, `Register-SSOOIDCClient`, `Start-SSOOIDCDeviceAuthorization`, and `New-SSOOIDCToken`) to prevent live AWS API calls during Unit_Test execution.
2. WHEN `Find-CFNStackErrors` is invoked, THE Test_Suite SHALL verify that stacks with non-null `StackStatusReason` are included in the output stream and stacks with null `StackStatusReason` are excluded.
3. WHEN `Set-AWSProfileWithMFA` is invoked with a valid profile and token code, THE Test_Suite SHALL verify that `Get-STSSessionToken` is called with the `-SerialNumber` matching the value returned by `Get-IAMMFADevice` and the `-TokenCode` matching the supplied token.
4. WHEN `Set-AWSEnv` is invoked with `-WhatIf`, THE ShouldProcess_Test SHALL verify that no environment variables are modified.
5. WHEN `Remove-ExpiredAWSProfiles` encounters a profile with an `ExpiredToken` error, THE Test_Suite SHALL verify that `Remove-AWSCredentialProfile` is called for that profile.
6. WHEN `Start-MultiStackDriftDetection` is invoked with stack names, THE Test_Suite SHALL verify that `Start-CFNStackDriftDetection` is called for each stack whose status is not in `ROLLBACK_COMPLETE`, `DELETE_FAILED`, or `ROLLBACK_FAILED`.
7. WHEN `Start-MultiStackDriftDetection` encounters a stack in `ROLLBACK_COMPLETE`, `DELETE_FAILED`, or `ROLLBACK_FAILED` status, THE Test_Suite SHALL verify that `Start-CFNStackDriftDetection` is not called for that stack.
8. WHEN `Get-AWSAccountListOfDriftedResources` is invoked, THE Test_Suite SHALL verify that only resources with drift status `MODIFIED` or `DELETED` are returned and resources with drift status `NOT_MODIFIED` or `IN_SYNC` are excluded.
9. WHEN `Get-AWSObjectCount` is invoked for a single region, THE Test_Suite SHALL verify that the output object contains `Region`, `StackCount`, `VPCCount`, `EC2Count`, `BucketCount`, `LambdaCount`, and `ScanOk` properties.
10. WHEN `Use-AssumedRole` is invoked with a valid role name, THE Test_Suite SHALL verify that `Use-STSRole` is called with the role ARN resolved from `Get-IAMRole` and that `Set-AWSCredential` is called with the returned temporary credentials.
11. WHEN `Update-SSOCredentialList` is invoked, THE Test_Suite SHALL verify that `Register-SSOOIDCClient` and `Start-SSOOIDCDeviceAuthorization` are called to initiate device authorization, and that `New-SSOOIDCToken` is called to obtain the access token, and that credentials are written to the credential file.
12. WHEN `Clear-S3Bucket` is invoked with `-WhatIf`, THE ShouldProcess_Test SHALL verify that `Remove-S3Object` is not called.
13. WHEN `Remove-ExpiredAWSProfiles` encounters a profile whose `Get-STSCallerIdentity` call succeeds, THE Test_Suite SHALL verify that `Remove-AWSCredentialProfile` is not called for that profile.

### Requirement 4: CloudFormation Function Tests

**User Story:** As a developer, I want unit tests for CloudFormation template processing functions, so that I can validate stack operations without deploying to AWS.

#### Acceptance Criteria

1. WHEN `New-CFNStackFromDirectory` is invoked with a valid stack directory containing `template.template` and `parameters.json`, THE Test_Suite SHALL verify that the template is uploaded to S3, `Test-CFNTemplate` is called for validation, and `New-CFNStack` is called with the StackName, TemplateURL, Parameters, Capabilities, and Tags arguments.
2. WHEN `Test-CFNStackFromDirectory` is invoked with a valid stack directory, THE Test_Suite SHALL verify that the template body is read from the `template.template` file and passed to the `Test-CFNTemplate` cmdlet.
3. WHEN `Out-CFNStackInfo` is invoked, THE Test_Suite SHALL verify that stack information is exported as `template.template`, `parameters.json`, `tags.json`, `capabilities.json`, `outputs.json`, and `stack.json` within the `{AccountID}/{Region}/{StackName}` directory structure.
4. WHEN `Update-CFNStackFromDirectory` is invoked with `-WhatIf`, THE ShouldProcess_Test SHALL verify that neither `New-CFNChangeSet` nor `Start-CFNChangeSet` is called.
5. WHEN `New-CFNStackDirectory` is invoked with a StackName and TemplateBody, THE Test_Suite SHALL verify that a subdirectory named after the StackName is created and a `template.template` file is written with the provided TemplateBody content.
6. WHEN `Edit-CFTTEbsVolumes` is invoked with a template containing EBS volume definitions using the OldVolumeType value, THE Test_Suite SHALL verify that all occurrences of OldVolumeType are replaced with NewVolumeType in the resulting template body and that a change set is created via `New-CFNChangeSet`.
7. IF a CloudFormation function is invoked with a stack directory that does not contain the required `template.template` file, THEN THE Test_Suite SHALL verify that the function writes an error indicating the template file was not found.
8. IF `New-CFNStackFromDirectory` is invoked and the template parameter count does not match the `parameters.json` entry count, THEN THE Test_Suite SHALL verify that the function throws an error indicating a parameter count mismatch.

### Requirement 5: Git Function Tests

**User Story:** As a developer, I want unit tests for Git-related functions, so that I can validate hook installation and signature checking without modifying real repositories.

#### Acceptance Criteria

1. WHEN `Test-CommitSignatures` is invoked with `-Count 1` in a temporary git repository whose most recent commit has a valid signature (status 'G'), THE Test_Suite SHALL verify that the function returns `$true`.
2. WHEN `Test-CommitSignatures` is invoked with `-Count 1` in a temporary git repository whose most recent commit has no signature (status 'N'), THE Test_Suite SHALL verify that the function returns `$false`.
3. WHEN `Test-CommitSignatures` is invoked in a temporary directory that is not a git repository, THE Test_Suite SHALL verify that the function writes a terminating error via `Write-Error` containing the text "Not in a git repository" and returns `$false`.
4. WHEN `Install-GitHooks` is invoked in a temporary git repository that contains a `.githooks/pre-commit` source file, THE Test_Suite SHALL verify that the file is copied to `.git/hooks/pre-commit` and the destination file content matches the source.
5. WHEN `Install-GitHooks` is invoked without `-Force` in a temporary git repository where `.git/hooks/pre-commit` already exists, THE Test_Suite SHALL verify that `Read-Host` is called to prompt the user before overwriting.
6. IF the source hook file `.githooks/pre-commit` does not exist in the repository root, THEN THE Test_Suite SHALL verify that `Install-GitHooks` writes an error via `Write-Error`, does not create a `.git/hooks/pre-commit` file, and returns without throwing a terminating exception.
7. WHEN `Install-GitHooks` is invoked with `-Force` in a temporary git repository where `.git/hooks/pre-commit` already exists, THE Test_Suite SHALL verify that the existing hook is overwritten without prompting the user.
8. WHEN `Install-GitHooks` is invoked without `-Force` and the user responds with a value other than 'y' to the overwrite prompt, THE Test_Suite SHALL verify that the existing `.git/hooks/pre-commit` file is preserved unchanged.

### Requirement 6: Private Helper Function Tests

**User Story:** As a developer, I want unit tests for private helper functions that contain reusable logic, so that I can catch regressions in shared internal behavior.

#### Acceptance Criteria

1. THE Test_Suite SHALL maintain the existing `New-AWSParamSplat` tests in `tests/New-AWSParamSplat.Tests.ps1` covering filtering behavior, all 8 supported AWS parameter names, and null/empty value exclusion.
2. WHEN `CFNStackDirectoryInfo.ValidateForDeploy` is called with a directory missing one or more of the required files (`template.template`, `parameters.json`), THE Test_Suite SHALL verify that the returned array contains exactly the names of the missing files.
3. WHEN `CFNStackDirectoryInfo.ValidateForDeploy` is called with a directory containing all required files, THE Test_Suite SHALL verify that the returned array is empty.
4. WHEN `CFNStackDirectoryInfo.GetStackExportPath` is called with a RootPath, AccountID, Region, and StackName, THE Test_Suite SHALL verify that the returned path equals the platform-appropriate join of `RootPath/AccountID/Region/StackName`.
5. WHEN `Get-CFNContext` is called with an explicit Region parameter, THE Test_Suite SHALL verify that the returned hashtable contains `AccountID`, `Region`, and `TemplateBucket` keys, and that `Region` equals the explicitly provided value.
6. WHEN `Get-CFNContext` is called without a Region parameter, THE Test_Suite SHALL verify that the returned hashtable uses the value from `Get-DefaultAWSRegion` as the `Region` key.

### Requirement 7: Help Discoverability Tests

**User Story:** As a developer, I want automated tests that verify every exported function has discoverable comment-based help, so that the help gate passes before release.

#### Acceptance Criteria

1. THE Test_Suite SHALL include a Help_Test for each function listed in the module manifest `FunctionsToExport` array.
2. WHEN `Get-Help` is called for an exported function, THE Help_Test SHALL verify that the `Synopsis` property is not null, not empty, and does not match the auto-generated pattern (function name repeated as synopsis).
3. WHEN `Get-Help` is called for an exported function, THE Help_Test SHALL verify that at least one example is present in the help output.
4. WHEN `Get-Help` is called for an exported function, THE Help_Test SHALL verify that the `Description` property is not null or empty.

### Requirement 8: Error Path and Edge Case Coverage

**User Story:** As a developer, I want tests that exercise error paths and edge cases, so that I can ensure the module handles failures gracefully.

#### Acceptance Criteria

1. WHEN an AWS cmdlet mock throws an exception during `Invoke-ScriptMultiAccountRegion` execution, THE Test_Suite SHALL verify that a warning is written containing the profile name, region, and the exception message from the mock.
2. IF `Invoke-ScriptMultiAccountRegion` is invoked without a `-ProfileName` parameter and no default profile is configured (no `$StoredAWSCredentials` and no 'default' profile in credential store), THEN THE Test_Suite SHALL verify that a non-terminating error is written indicating no profile was found.
3. WHEN `Invoke-ScriptMultiAccountRegion` encounters an authentication failure (Get-STSCallerIdentity throws) for a profile, THE Test_Suite SHALL verify that a warning is written containing the failing profile name and that subsequent profiles in the list are still processed.
4. WHEN `Clear-AuthenticodeSignature` is invoked with a file that has no signature block, THE Test_Suite SHALL verify that the file content remains byte-for-byte unchanged and no error is written.
5. IF `Set-FileSignature` receives a certificate whose Issuer property does not start with "CN=Digicert" or "CN=Sect", THEN THE Test_Suite SHALL verify that a terminating error is thrown with the message "No Timestamp server could be set, aborting."
6. WHEN `Get-AWSObjectCount` encounters an AWS API error for a region, THE Test_Suite SHALL verify that the output object for that region has `ScanOk` set to `$false` and count properties set to empty strings.

### Requirement 9: Code Coverage Measurement

**User Story:** As a developer, I want the test suite to measure and report code coverage, so that I can track progress toward the 80% coverage target.

#### Acceptance Criteria

1. THE Test_Suite SHALL configure Pester code coverage in `tests/pester.config.ps1` to target all `.ps1` and `.psm1` files under `src/CharlandCustomizations/`.
2. WHEN `Invoke-Pester` is run with the code coverage configuration, THE Test_Runner SHALL produce a coverage report to the console showing the percentage of lines covered per source file and the overall line coverage percentage.
3. IF the overall line coverage falls below 80%, THEN THE Test_Runner SHALL report a coverage failure indicating the current percentage and the 80% threshold.
4. WHEN `Invoke-Pester` is run with code coverage enabled, THE Test_Runner SHALL export a JaCoCo-format XML coverage report to `tests/coverage/coverage.xml`.

### Requirement 10: Test Execution and Tagging

**User Story:** As a developer, I want tests tagged by category so that I can run subsets of the test suite quickly during development.

#### Acceptance Criteria

1. THE Test_Suite SHALL tag all unit tests with the `Unit` tag at the `Describe` block level.
2. THE Test_Suite SHALL tag all help discoverability tests with the `Help` tag at the `Describe` block level.
3. THE Test_Suite SHALL tag all integration tests with the `Integration` tag at the `Describe` block level.
4. WHEN `Invoke-Pester -Tag Unit` is run, THE Test_Runner SHALL execute only tests tagged `Unit` and SHALL NOT execute tests tagged `Integration` or `Help`.
5. THE Test_Suite SHALL complete all `Unit` tagged tests within 60 seconds when run on a machine with at least 4 CPU cores and 8 GB RAM.
6. WHEN `Invoke-Pester` is run without a `-Tag` or `-ExcludeTag` parameter, THE Test_Runner SHALL execute all tests tagged `Unit` and `Help` but SHALL NOT execute tests tagged `Integration`.
